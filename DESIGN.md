# calyra-ai Design System

本文件是 calyra-ai 的前端设计执行规范。它补充 `AGENTS.md` 中的全站前端设计原则，用于约束页面重构、组件新增和视觉细节统一。

## 设计方向

calyra-ai 的视觉语言应保持克制、清晰、稳定，参考 Google / Material 3 的信息层级、状态表达和响应式秩序，同时吸收 Apple 式的留白、精致感和低噪音界面。

不要把“像 Apple / Google”理解成复制某个系统皮肤。落地标准是：

- 页面目标清楚，用户一眼知道当前页面是什么。
- 操作路径清楚，用户知道下一步能做什么。
- 状态反馈清楚，用户知道生成、保存、失败、支付、积分变化等状态。
- 视觉层级清楚，标题、说明、操作、结果、辅助信息各司其职。
- 细节统一，按钮、输入框、卡片、徽标、间距、圆角、阴影不要各页一套。

## 禁止项

- 不做模板感 landing page。
- 不使用大面积渐变背景、装饰性 orb、bokeh、无意义发光块。
- 不堆叠过多卡片，不把页面 section 做成一层套一层的卡片。
- 不使用低信息密度 hero，不为了视觉冲击牺牲移动端首屏效率。
- 不使用过重阴影、过度玻璃拟态、过度动效。
- 不让页面被单一色系支配，尤其避免整页紫色、蓝紫渐变、深蓝灰、米色或咖啡色调。
- 不在功能页使用营销页式大标题、大插画和过宽留白。

## 页面类型

### 公开增长页

适用：`/`、`/pricing`、`/song/[id]`、公开内容页、未来 SEO 词簇页。

设计目标：

- SEO 闭环第一，视觉第二但必须精致。
- 保证 SSR 可索引文本、语义 heading、结构化数据、内链、分享预览和 CTA。
- 首屏必须说明产品价值，并展示清晰主 CTA。
- 视觉不能隐藏核心内容，不能依赖 client-only 动效表达关键信息。
- 移动端首屏要能看到页面主题和至少一个主要行动入口。

### 产品功能页

适用：`/create`、`/dashboard`、音频生成流程、私有报告页。

设计目标：

- 用户任务效率第一，界面像工具，不像营销页。
- 当前状态、下一步操作、积分/成本影响、生成进度和错误恢复方式必须清楚。
- 信息密度可以更高，但分区要明确，扫描成本要低。
- 表单、步骤、结果、历史记录和操作区应保持稳定布局，不因状态变化频繁跳动。

### 认证和设置页

适用：`/sign-in`、`/sign-up`、`/forgot-password`、账户相关页面。

设计目标：

- 信任感、简洁和少干扰第一。
- 降低视觉噪音，突出登录、注册、找回密码、账户设置等核心动作。
- 不使用复杂背景、重装饰和多余营销信息。
- 错误、成功、邮箱验证、加载状态要明确而克制。

## 布局规范

- 公共增长页内容容器优先使用 `max-w-6xl` 或 `max-w-7xl`。
- 阅读型正文区域宽度应更窄，避免长行文本影响阅读。
- 产品功能页可使用更宽布局或 app shell，但操作区、状态区和结果区要有清晰分组。
- section 使用全宽分段 + 内部容器，不把整个 section 包进浮动大卡片。
- 避免嵌套卡片。卡片只用于独立重复项、表单块、结果块、弹窗或真正需要边界的工具区域。
- 桌面端常规 section 垂直间距建议 `py-16` 到 `py-24`。
- 移动端常规 section 垂直间距建议 `py-10` 到 `py-14`。
- 固定格式元素，如播放器、生成面板、价格卡、歌曲卡、工具栏，要定义稳定尺寸或响应式约束，避免 hover、loading、长文本导致布局跳动。

## 字体规范

- 每页只保留一个主 `h1`。
- 公共增长页 `h1` 可以更强，通常为 `text-4xl md:text-6xl`。
- 产品功能页 `h1` 应克制，通常为 `text-2xl md:text-3xl`。
- section 标题通常为 `text-2xl md:text-4xl`。
- 工具面板、卡片和表单内部标题应更小，通常为 `text-base` 到 `text-xl`。
- 正文优先使用 `text-sm` 或 `text-base`。
- 辅助说明使用 `text-xs` 或 `text-sm`，颜色弱化但必须可读。
- 不使用负字间距。避免用 viewport 宽度动态缩放字体。
- 长标题和按钮文案必须在移动端可换行或收缩，不能溢出容器。

## 颜色规范

- 以中性背景和清晰前景为主，强调色只用于关键操作、品牌识别和状态提示。
- 背景层级优先通过 `background`、`muted`、`border`、轻微色块区分，不依赖重阴影。
- 渐变只能少量用于强调，不能作为大面积装饰背景。
- 状态色必须语义化：
  - success：成功、完成、已发布。
  - warning：风险、等待处理、积分影响提醒。
  - error：失败、阻断、危险操作。
  - info：说明、处理中、系统提示。
- 同一种状态在全站使用同一种视觉语义，不要一页一个颜色含义。

## 组件基础规范

- 优先复用现有 `shadcn/ui` 组件，再做项目级封装。
- 新增组件文件使用 kebab-case。
- 组件样式优先使用 Tailwind token 和已有 variant，不写孤立的一次性视觉规则。
- 重复出现两次以上的页面结构，应考虑抽成项目级小组件。
- 组件默认状态、hover、focus、disabled、loading、error 状态必须完整。
- 图标优先使用项目已有图标库或 lucide 图标，不手写装饰性 SVG。
- 图标按钮必须有明确语义，必要时提供 tooltip 或 `aria-label`。

## 小组件规范

### PrimaryButton

用途：页面或区域内最重要的主行动，例如生成歌曲、开始创作、购买计划、保存关键结果。

规范：

- 同一视图内主按钮数量要少，通常只有一个真正的 primary。
- 高度统一使用 `h-11` 或 `h-12`。
- 文案使用明确动词，不使用模糊表达。
- loading 时保持宽度稳定，避免按钮文字变化导致布局跳动。

### SecondaryButton

用途：次级行动，例如查看详情、返回、复制、预览、切换方案。

规范：

- 视觉权重明显低于 PrimaryButton。
- 不与 primary 使用相同背景强调。
- 与 primary 并排时，顺序和间距保持一致。

### IconButton

用途：单一图标动作，例如播放、暂停、复制、下载、删除、更多操作。

规范：

- 使用固定正方形尺寸，例如 `h-9 w-9` 或 `h-10 w-10`。
- 必须有 `aria-label`。
- 对不够直观的图标提供 tooltip。
- 危险操作不能只靠颜色表达，必要时加确认。

### PageHeader

用途：页面顶部标题区。

规范：

- 包含页面标题、简短说明和主要操作。
- 公共页可以更有品牌表达，产品页保持紧凑。
- 不在同一页重复多个 PageHeader。
- 移动端主操作应靠近标题区，不能被挤到很远。

### SectionHeader

用途：页面分区标题。

规范：

- 层级弱于 PageHeader。
- 包含标题和可选说明，说明不宜过长。
- 同一页面的 SectionHeader 间距、字号、说明颜色保持一致。

### FormField

用途：表单字段标准结构。

规范：

- 结构固定为 label、control、hint、error。
- label、hint、error 的字号和颜色统一。
- 错误信息放在字段附近，不集中堆到页面顶部。
- 必填、禁用、只读状态要清楚。

### EmptyState

用途：空列表、无历史记录、无生成结果、无搜索结果。

规范：

- 必须包含一句解释和一个可执行动作。
- 不允许只有“暂无数据”。
- 图标或插图可选，但不能成为主要内容。
- 产品页空状态应直接引导用户继续任务。

### StatusBadge

用途：表达对象状态，例如 ready、processing、failed、private、public、paid。

规范：

- 只表达状态，不承担按钮功能。
- 颜色语义固定，不能混用。
- 文案短，避免长句。
- 与表格、卡片、详情页中的状态表达保持一致。

### CreditCostNotice

用途：所有涉及积分消耗、冻结、返还或付费影响的动作前提示。

规范：

- 必须出现在用户确认消耗前。
- 明确显示消耗数量、冻结/扣除时机、失败是否返还。
- 视觉要明显但不恐吓。
- 与生成按钮位置接近，避免用户忽略成本。

### ProgressBlock

用途：歌词生成、音频生成、封面生成、上传、渲染等长流程。

规范：

- 显示当前步骤、整体状态和可恢复错误。
- 文案要具体，不只显示 “loading”。
- 长流程要避免界面空白，可以显示阶段说明或最近结果。
- 失败时提供重试、返回编辑或查看已生成内容的路径。

### InfoRow

用途：详情页、账单、歌曲信息、报告摘要中的键值信息。

规范：

- 左侧标签，右侧内容。
- 标签宽度和对齐保持稳定。
- 长内容允许换行，但不能破坏整体布局。
- 不用 InfoRow 展示长段正文。

### SectionCard

用途：真正需要边界的单块内容，例如表单块、生成结果块、价格计划、歌曲卡。

规范：

- 圆角保持克制，优先 `rounded-lg` 或更小。
- 阴影轻，优先边框和背景层级。
- 不把 SectionCard 套在另一个 SectionCard 里。
- 卡片内标题、说明、操作按钮要有稳定间距。

### StatusBanner

用途：页面级提示，例如支付成功、生成失败、邮箱未验证、系统维护。

规范：

- 放在与上下文相关的位置，不随意浮在页面顶部。
- 状态色语义清楚。
- 包含具体说明和必要操作。
- 可关闭提示要有明确关闭按钮和可访问名称。

### GenerationPanel

用途：`/create` 和生成流程的核心输入/控制面板。

规范：

- 输入、风格、积分提示、主 CTA、进度反馈应在同一区域内形成连续任务流。
- 高级选项默认折叠或弱化，避免阻断主流程。
- 生成中禁用会造成冲突的操作，但保留可理解的返回或查看路径。
- 错误恢复路径必须清楚，例如重试、修改输入、联系客服或查看积分。

## 页面落地要求

### 首页

- 首屏表达产品价值、核心结果和主 CTA。
- 避免低信息密度大 hero。
- 必须保留 SEO 可读内容和内链。
- 视觉上体现 AI 音乐创作，但不要使用纯装饰性背景代替产品信息。

### Create 页面

- 输入故事、选择风格、查看积分消耗、开始生成、查看进度必须顺畅。
- 主 CTA 附近必须显示 CreditCostNotice。
- 生成流程必须使用统一的 ProgressBlock 或等价结构。
- 错误必须可恢复，不能只弹 toast。

### Pricing 页面

- 价格、权益、限制、CTA、常见问题要可比较。
- 推荐计划可以强调，但不能让其他计划不可读。
- 支付和积分相关说明必须清楚。
- 移动端计划卡顺序和 CTA 要稳定。

### Dashboard 页面

- 以管理效率为主，优先展示最近作品、状态、筛选、继续操作。
- 空状态必须引导创建第一首歌。
- 列表、卡片、状态徽标和操作按钮保持一致。

### Song 公开页

- 播放、封面、标题、风格、分享和 CTA 是核心。
- 公开页必须服务传播和转化，但不能遮挡播放体验。
- 分享卡片和页面视觉要一致，避免像两个产品。

## 执行流程

改 UI 时遵循以下顺序：

1. 先判断页面类型：增长页、产品页、认证/设置页。
2. 读取现有组件和相邻页面风格，优先统一而不是另起一套。
3. 先修结构和层级，再修颜色和装饰。
4. 优先抽取重复小组件，再处理单页细节。
5. 检查移动端首屏、按钮文案、长文本、loading、empty、error 状态。
6. 完成后运行项目既有验证命令，至少执行 `pnpm build`，除非本次只改文档。

## 判断标准

一次 UI 修改完成后，应能回答：

- 用户是否能在 3 秒内理解页面目的？
- 主行动是否清楚且没有被其他元素抢焦点？
- 积分、支付、生成、失败等关键状态是否明确？
- 同类组件是否保持同样高度、圆角、间距、颜色和状态？
- 移动端是否比桌面端更紧凑但不拥挤？
- 页面是否避免了模板感和过度装饰？
